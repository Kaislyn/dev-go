### 为什么大厂一定要用 Viper？

在之前的代码里，我们将端口号写死了 `:8080`，把日志大小写死了 `MaxSize: 10`。在真实的开发中，我们会有 **本地环境 (Local)**、**测试环境 (Test)** 和 **生产环境 (Prod)**。每个环境的数据库密码、端口号、日志路径都不一样。

如果每次切换环境都要去改 Go 源码然后重新编译，那是灾难级的。**标准做法是：代码编译后只有一份，通过读取外部的配置文件（如 `.yaml`）来决定程序的运行状态。**

`Viper` 是 Go 语言生态里绝对的配置管理“一哥”（连 Kubernetes 都在用它）。它支持 JSON、TOML、YAML 等多种格式。今天我们采用云原生最流行的 **YAML** 格式来进行实战。

---

### 第一步：安装 Viper
在终端执行以下命令拉取依赖：
```bash
go get -u github.com/spf13/viper
```

### 第二步：创建配置文件
在你的项目根目录下（也就是和 `main.go` 同级的地方），新建一个文件叫 `config.yaml`。
把我们之前写死的配置全部抽离到这里面：

```yaml
# config.yaml
app:
  name: "kaisgo-backend"
  mode: "debug"   # 开发模式 debug，生产模式 release
  port: 8080

log:
  filename: "./logs/app.log"
  max_size: 10     # MB
  max_backups: 5
  max_age: 30      # 天
  level: "debug"   # 输出的日志级别
```

### 第三步：定义对应的 Go 结构体 (核心)
Viper 最强大的地方在于，它可以把上面那个 YAML 文件，**一键反序列化（Unmarshal）** 到 Go 的结构体里。

在项目里新建一个 `config.go` 文件（或者直接写在 `main.go` 上方），定义以下结构体。
**注意重点：** 这里用的标签不是 `` `json:"..."` ``，而是 `` `mapstructure:"..."` ``，这是 Viper 底层依赖的反序列化库专属标签。

```go
package main

import (
	"fmt"
	"github.com/spf13/viper"
)

// 全局配置变量，以后在代码任何地方调用 Conf.App.Port 就能拿到配置
var Conf *Config

type Config struct {
	App AppConfig `mapstructure:"app"`
	Log LogConfig `mapstructure:"log"`
}

type AppConfig struct {
	Name string `mapstructure:"name"`
	Mode string `mapstructure:"mode"`
	Port int    `mapstructure:"port"`
}

type LogConfig struct {
	Filename   string `mapstructure:"filename"`
	MaxSize    int    `mapstructure:"max_size"`
	MaxBackups int    `mapstructure:"max_backups"`
	MaxAge     int    `mapstructure:"max_age"`
	Level      string `mapstructure:"level"`
}

// ==========================================
// 初始化 Viper
// ==========================================
func InitConfig() {
	viper.SetConfigFile("config.yaml") // 指定配置文件路径
	
	// 1. 读取配置文件
	if err := viper.ReadInConfig(); err != nil {
		panic(fmt.Errorf("读取配置文件失败: %v", err))
	}

	// 2. 将配置解析到全局变量 Conf 中
	if err := viper.Unmarshal(&Conf); err != nil {
		panic(fmt.Errorf("解析配置文件失败: %v", err))
	}

	fmt.Println("Viper: 配置加载成功！当前端口:", Conf.App.Port)
}
```

### 第四步：改造原有的 Logger 和 Main 函数
现在我们的配置已经活了，接下来就是把上一节的 `InitLogger` 和 `main` 函数里的“死代码”替换掉。

回到你的 `main.go`：

```go
// ... 之前的 import 保持不变 ...

func InitLogger() *zap.Logger {
	// ！！看这里：以前写死的 10, 5, 30，现在全部从 Conf.Log 里读取！！
	logWriter := &lumberjack.Logger{
		Filename:   Conf.Log.Filename,
		MaxSize:    Conf.Log.MaxSize,
		MaxBackups: Conf.Log.MaxBackups,
		MaxAge:     Conf.Log.MaxAge,
		Compress:   true,
	}

	encoderConfig := zap.NewProductionEncoderConfig()
	encoderConfig.EncodeTime = zapcore.ISO8601TimeEncoder
	encoderConfig.EncodeLevel = zapcore.CapitalLevelEncoder

	encoder := zapcore.NewConsoleEncoder(encoderConfig)
	fileSyncer := zapcore.AddSync(logWriter)
	consoleSyncer := zapcore.AddSync(os.Stdout)
	
	core := zapcore.NewCore(
		encoder,
		zapcore.NewMultiWriteSyncer(fileSyncer, consoleSyncer),
		zap.DebugLevel, 
	)

	return zap.New(core, zap.AddCaller())
}

// ... ZapLogger 和 ZapRecovery 保持不变 ...

func main() {
	// 1. 第一步永远是先初始化配置！
	InitConfig()

	// 2. 根据配置初始化日志
	logger := InitLogger()
	defer logger.Sync()

	// 3. 根据配置设置 Gin 的运行模式 (Release 或 Debug)
	gin.SetMode(Conf.App.Mode)
	
	r := gin.New()
	r.Use(ZapLogger(logger), ZapRecovery(logger))

	r.GET("/ping", func(c *gin.Context) {
		Success(c, "pong")
	})

	// ！！看这里：以前写死的 ":8080"，现在拼接字符串读取配置！！
	addr := fmt.Sprintf(":%d", Conf.App.Port)
	logger.Info("Server is running...", zap.String("addr", addr))
	r.Run(addr)
}
```
