# 核心后端组件：JWT (JSON Web Token)

## 一、为什么大厂都在用 JWT？(告别 Session)
在传统的 Web 开发中，用户登录后，服务器会在内存里存一段 Session（会话），然后给浏览器发一个 Cookie。

**传统 Web：Session + Cookie 交互流程:**

1. 用户登录成功后，**服务器创建 Session 存储用户信息**，并生成唯一 `SessionID`。
2. 服务器通过 `Set-Cookie` 响应头，将 `SessionID` 下发给浏览器，浏览器自动保存为 Cookie。
3. 后续每次请求，浏览器**自动在请求头携带该 Cookie**。
4. 服务器通过 `SessionID` 查找对应 Session，完成用户身份识别。

**核心痛点：** 状态无法共享。如果大厂有 1000 台服务器，用户在 A 服务器登录，下次请求被负载均衡分配到 B 服务器，B 服务器内存里没有这个 Session，用户就会被迫重新登录。

**具体**：
- 服务端**有状态**，需本地存储 Session；
- 分布式集群下，多服务器 Session 不互通，必须额外做共享（如 Redis），架构复杂。

**JWT 的降维打击（无状态 Stateless）：**
服务器根本不记你登没登录！你登录成功后，服务器用**哈希签名算法**给你颁发一张“包含你身份信息的 VIP 通行证”（这就是 JWT）。以后你每次请求 API 只要带着这张通行证，服务器用密钥验一下签名：如果没被篡改过，就直接放行。彻底解决了分布式系统下的认证难题。

## 二、JWT 标准结构

JWT 由 `.` 分割的三段字符串组成：
1. **Header（头部）**：记录签名算法（如 HS256）与令牌类型。
2. **Payload（载荷）**：存放**非敏感**的用户业务信息（如 UserID、Role）以及标准声明（如过期时间 `exp`、签发时间 `iat`）。*注意：此部分仅做 Base64Url 编码，不加密，任何人可解码查看。*
3. **Signature（签名）**：核心防伪标识。由服务器独占的密钥结合前两部分计算生成。

## 三、核心运行原理 (面试必答)
### 1. 令牌签发（登录成功时）
服务器拼装 Header + Payload → 使用**服务端密钥 + 签名算法**生成不可伪造的数字签名 → 拼接成完整 JWT 字符串返回给前端。

### 2. 令牌解析/校验（请求受保护接口时）
1. **拆分**：将传入的 JWT 按 `.` 拆分为 Header、Payload、原始签名。
2. **验算法**：校验 Header 中的算法是否符合服务端预期，防止降级攻击。
3. **重新签名**：服务器使用自身密钥对前两部分**重新计算一次新签名**。
4. **比对验证**：
   * **新签名 == 原始签名**：合法令牌，未被篡改，校验通过。
   * **新签名 != 原始签名**：伪造或篡改令牌，直接报废，拒绝访问。

## 四、核心结论与安全红线
1. JWT 是**无状态**身份凭证，将存储压力转移给了客户端。
2. **校验的核心是比对签名，而不是比对令牌本身**。
3. 令牌一旦被篡改即永久失效，需要重新登录获取新令牌。
4. **安全红线 1**：Payload 绝对不能存放密码、身份证等敏感信息。
5. **安全红线 2**：必须设置合理的长短过期时间（如 2 小时）。
6. **安全红线 3**：服务器密钥（SecretKey）属最高机密，严禁硬编码，必须通过配置中心（如 Viper）读取。
