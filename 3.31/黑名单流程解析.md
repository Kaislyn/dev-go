# JWT + Redis 黑名单模式 

## 一、先回顾：没有 Redis 的纯 JWT 有什么问题？

**纯 JWT 的工作方式**：
1. 用户登录成功 → 服务端生成一个 JWT（含用户ID、过期时间等）
2. 服务端**不保存**这个 token
3. 之后用户每次请求带上 token，服务端只验证**签名是否正确 + 是否过期**
4. 验证通过就放行

**问题**：一旦 token 签发出去，服务端就**再也无法主动让它失效**。  
比如：
- 用户点“退出登录” → 客户端删了 token，但 token 本身依然有效。如果有人偷了这个 token，照样能调用接口。
- 管理员想封禁一个用户 → 只要用户的 token 没过期，他依然可以访问。
- 用户修改密码 → 旧 token 还能用，这不合理。

**本质**：服务端对已签发的 token 失去了**撤销能力**。

---

## 二、黑名单模式的核心思想

**一句话**：  
> 维护一个“坏 token 列表”（黑名单），每次请求不仅要验证 JWT 本身，还要检查这个 token 是否在黑名单里。

**黑名单存在哪里**？  
→ Redis（因为需要高速查询，且 token 有剩余有效期，可以利用 Redis 的自动过期功能）。

**黑名单里存什么**？  
→ 通常存 token 的唯一标识（比如 JWT 标准字段 `jti`，或者 `用户ID+过期时间` 的组合），并设置**过期时间 = 该 token 的剩余有效期**。  
这样 token 原本过期后，黑名单里的记录也自动删除，不会无限堆积。

---

## 三、黑名单模式完整流程（分步骤）

### 场景假设
- 用户 A 登录成功，获得一个有效期为 7 天的 JWT。
- 用户在第二天主动登出。
- 管理员在第三天发现用户违规，强制踢下线。
- 用户在第四天修改密码。

下面看每个环节如何处理。

---

### 流程 1：用户登录（签发 token）

**操作**：
1. 验证用户名/密码正确。
2. 生成 JWT，其中至少包含：
   （user_id、exp、jti 都在载荷里）
   - `sub` 或 `user_id`：用户ID
   - `exp`：过期时间（比如 7 天后）
   - `jti`：一个唯一字符串（UUID），用于标识这个 token
3. **此时不往 Redis 里存任何东西**（保持无状态）。
4. 把 token 返回给客户端。

**关键点**：  
- 登录时**不写黑名单**，也不写白名单。黑名单初始为空。

---

### 流程 2：用户正常请求（中间件验证）

每次请求受保护接口时：

1. 从 `Authorization` 头取出 token。
2. 验证 JWT 签名和有效期（是否被篡改、是否过期）。
   - 如果验证失败 → 直接返回 401。
3. **验证通过后，立即检查 Redis 黑名单**：
   - 用 `jti` 作为 key，例如 `blacklist:jti值`。
   - 如果 Redis 中存在这个 key → 说明 token 已被拉黑 → 返回 401。
   - 如果不存在 → 放行。
4. 从 token 中取出 `user_id` 等，存入 Gin 的上下文，供后续 handler 使用。

**关键点**：  
- 大部分请求都会查一次 Redis，但 Redis 很快，可以接受。
- 只有未被拉黑的 token 才能通过。

---

### 流程 3：用户主动登出（加入黑名单）

用户点击“退出登录”，前端把 token 发到后端的 `/logout` 接口。

**后端操作**：
1. 从请求中拿到 token。
2. 解析出 `jti` 和 `exp`（过期时间戳）。
3. 计算剩余有效时间：`ttl = exp - 当前时间戳`（单位：秒）。
4. 将 `blacklist:jti` 写入 Redis，并设置过期时间为 `ttl`。
   - 例如：`SET blacklist:abc123 1 EX 3600`（假设还剩1小时过期）
5. 返回“登出成功”。

**效果**：  
- 这个 token 立即失效。即使用户客户端还存着它，再次请求时会被中间件拦截。
- 等到 token 原本的过期时间到了，Redis 里的黑名单记录也会自动删除，节省空间。

---

### 流程 4：管理员强制踢人（批量加入黑名单）

管理员封禁某个用户，要求该用户当前所有 token 立即失效。

**难点**：  
一个用户可能在多个设备登录，生成了多个不同的 token（每个 token 有自己的 `jti`）。  
我们需要知道该用户**所有还在有效期内的 jti**。

**常用做法**：  
在**登录时**额外记录一个“用户 ↔ jti 的关联”到 Redis。  
（注意：这不是黑名单本身，而是一个辅助映射，用于后续批量拉黑）

#### 4.1 改造登录流程（增加关联记录）
- 生成 JWT 后，除了返回 token，还要执行：
  ```redis
  SADD user:123:jtis "jti值1" "jti值2" ...
  ```
  把这次签发的 `jti` 加入到该用户的 jti 集合中。
- 同时为这个集合设置一个过期时间（比如略长于 token 的最大有效期，或利用 token 过期后的惰性清理）。

#### 4.2 强制踢人流程
管理员调用 `/admin/kick?user_id=123`：

1. 从 Redis 中取出 `user:123:jtis` 集合，得到该用户所有活跃的 `jti`。
2. 对每一个 `jti`：
   - 从对应的 token 中获取剩余有效期（或者提前在登录时额外存储每个 jti 的过期时间）。
   - 将 `blacklist:jti` 写入 Redis，过期时间设为剩余有效期。
3. 删除或清空 `user:123:jtis` 集合（可选，因为黑名单已生效，且旧 token 很快过期）。
4. 返回“已踢下线”。

**效果**：  
该用户所有已签发的 token 立即失效，无法再访问任何接口。

---

### 流程 5：修改密码（同强制踢人）

用户修改密码后，应该让**所有旧的 token 失效**（包括自己的其他设备）。

流程与强制踢人完全一样：
- 获取该用户的所有 `jti`
- 批量加入黑名单
- （可选）删除关联集合

## 五、常见问题与解答

### Q1：每个请求都查 Redis，会不会慢？
- Redis 单次查询通常 < 1ms，对大多数业务可以接受。
- 如果实在追求极致性能，可以结合本地缓存（如将黑名单的 jti 缓存几秒），但复杂度上升。
- 通常先优化数据库查询，最后才考虑这点。

### Q2：如果用户登出后，又把同一个 token 拿来刷新怎么办？
- 刷新 token 接口同样需要检查黑名单。已经在黑名单中的 token，刷新请求也会被拒绝。
- 所以登出后，refresh token 也必须加入黑名单（如果你们用的是双 token 机制）。

### Q3：修改密码时，我怎么知道每个 jti 还剩多少时间？
- 方法一：在登录时，额外存储 `jti → 过期时间戳` 到 Redis，踢人时直接读取。
- 方法二：不存储，而是允许黑名单的过期时间**大于 token 实际剩余时间**（例如统一设置为 7 天）。缺点：黑名单会多留几天，但影响不大。
- 方法三：在踢人时，从 token 里解析 `exp`。这要求你能够拿到原始 token 字符串。对于强制踢人场景，你没有用户的 token，所以方法三不可行。因此推荐方法一。

### Q4：黑名单和白名单有什么区别？
- **黑名单**：默认所有合法 token 都放行，只有明确被撤销的才拦截。需要维护一个“坏的列表”。
- **白名单**：默认所有 token 都不放行，只有登录时明确存到 Redis 里的 token 才允许。需要维护一个“好的列表”。
- 黑名单性能更好（大多数请求不查或查一次无命中），白名单更严格（服务端完全掌控）。

---

## 六、总结（黑名单模式）

| 阶段       | 动作                             | Redis 操作                   |
| ---------- | -------------------------------- | ---------------------------- |
| 登录       | 生成 JWT，返回 token             | **无**                       |
| 正常请求   | 验证 JWT + 查黑名单              | `GET blacklist:jti`          |
| 登出       | 将 token 的 jti 加入黑名单       | `SET blacklist:jti 1 EX ttl` |
| 强制踢人   | 获取用户所有 jti，批量加入黑名单 | `SADD` + 循环 `SET`          |
| 修改密码   | 同上                             | 同上                         |
| token 过期 | 黑名单记录自动删除               | Redis TTL 机制               |

**核心收获**：  
> Redis 黑名单给了服务端“撤销已签发 JWT”的能力，解决了纯 JWT 无法主动失效的痛点。

下一步你可以学习**白名单模式**，它与黑名单思路相反，适用于需要服务端完全控制会话的场景。
