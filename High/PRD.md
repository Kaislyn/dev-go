# 📄 产品需求文档 (PRD)：高性能短链接分发系统

## 一、 项目背景与目标 (Why)
**痛点：** 在日常业务中，很多网页的 URL 非常长（比如带有几十个参数的商品详情页）。如果直接放在短信里发给用户，会超出字数限制；如果转成二维码，长链接生成的二维码会密密麻麻，极难扫出。

**目标：** 开发一个服务，将任意长链接转换成极短的固定格式链接（如 `http://t.cn/aB3x9Q`）。当用户在浏览器访问这个短链接时，系统能瞬间将其重定向（302 跳转）到原始的长链接。

## 二、 核心业务名词解释
* **长链接 (Original URL)**：用户输入的原始网址（如 `https://github.com/kaisgo/My-Code/commit/123456`）。
* **短码 (Short Code)**：系统计算出的一串 6~8 位的短字符串（如 `aB3x9Q`）。
* **短链接 (Short URL)**：域名 + 短码拼成的最终网址（如 `http://localhost:8080/aB3x9Q`）。

## 三、 核心功能模块 (What)

### 模块 1：用户鉴权中心 (Gin + JWT)
* **需求：** 系统不能让任何人无限量地生成短链接（防止恶意攻击消耗数据库）。只有注册并登录的用户才能使用生成功能。
* **功能点：**
  * 用户注册（账号、密码）。
  * 用户登录（校验密码，颁发 JWT Token）。
  * *扩展点：如果后期做大了，可以结合你刚刚学的 Redis 做“单设备互踢”。*

### 模块 2：短链接生成服务 (核心业务)
* **需求：** 登录用户提交一个长链接，系统返回一个短链接。
* **功能点：**
  * **发号器/哈希计算**：接收长链接，通过算法（如 MurmurHash 或 雪花算法）生成唯一的 6 位短码。
  * **碰撞处理**：如果两个不同的长链接算出了同一个短码，系统要有机制解决冲突。
  * **数据落库**：将 `短码`、`长链接`、`创建人ID`、`过期时间` 存入 MySQL。

### 模块 3：短链接跳转服务 (高并发重灾区)
* **需求：** 任何人（无需登录）在浏览器访问短链接，瞬间跳转到真实网页。
* **功能点：**
  * **302 重定向**：解析 URL 中的短码，去数据库查出对应的长链接，命令浏览器跳转。
  * **【架构高光】缓存加速**：跳转请求极速且海量，必须先查 Redis。Redis 没有才去查 MySQL，查完后把结果写回 Redis。
  * **【架构高光】防穿透拦截**：如果黑客用不存在的短码（如 `123456`）疯狂请求我们，怎么防止 MySQL 被瞬间查崩溃？（这里我们会引入“布隆过滤器”或“缓存空值”策略）。

### 模块 4：数据统计与监控 (异步协程秀操作)
* **需求：** 用户想知道自己生成的短链接被点击了多少次。
* **功能点：**
  * 每次短链接被访问时，不仅要跳转，还要记录点击次数。
  * **【架构高光】异步削峰**：跳转必须在几毫秒内完成，绝对不能因为“记录点击次数”拖慢跳转速度。所以记录点击量的逻辑，必须丢进 Go 语言的 **Channel**，让后台的 **Goroutine** 去慢慢处理（这就是大厂最爱的异步削峰）。

## 四、 核心业务流程图 (User Flow)

举例以下两个场景：

**场景 A（搞生产）：你作为博主，想在B站发个长长的淘宝推广链接。**
1. 你打开我们的系统，输入账号密码登录（带上 JWT）。
   1. 把超长淘宝链接贴进去，点击“生成”。
2. 后端 Gin 接收请求，算出一个短码 `K8mP2s`，存进 MySQL，告诉你短链接是 `http://127.0.0.1:8080/K8mP2s`。

**场景 B（搞消费）：你的粉丝在B站看到了这个短链接并点击了它。**
1. 粉丝的浏览器向我们的服务器发起 `GET http://127.0.0.1:8080/K8mP2s` 请求（不带 JWT）。
2. 我们的路由拿到 `K8mP2s`，去 Redis 找，如果找到了，直接取出长链接。
3. Gin 返回 HTTP 状态码 302 和 Location 头部（指向淘宝），粉丝瞬间被传送到淘宝。
4. 后台悄悄起一个 Goroutine，给你的“点击量”偷偷加 1。
